# 코덱스 실행 계획서(ExecPlan) 작성 규칙

이 문서는 코딩 에이전트가 실제로 동작하는 기능이나 시스템 변경을 끝까지 구현할 수 있도록 따라야 하는 실행 계획서(이하 "ExecPlan")의 작성 기준을 설명합니다. 독자는 이 저장소를 전혀 모르는 초보자라고 가정합니다. 독자가 가진 것은 현재 작업 트리와 당신이 작성한 ExecPlan 파일 하나뿐이며, 이전 계획에 대한 기억도 없고 외부 문맥도 없습니다.

## ExecPlan과 PLANS.md 사용 방법

실행 가능한 명세서인 ExecPlan을 작성할 때는 `PLANS.md`를 한 줄도 빠뜨리지 않고 따라야 합니다. 현재 문맥에 이 문서가 없다면 `PLANS.md` 전체를 다시 읽어 기준을 복원해야 합니다. 정확한 명세를 만들기 위해 관련 소스와 문서를 꼼꼼히 읽고, 필요하다면 다시 읽으십시오. 새 명세를 만들 때는 아래의 골격에서 시작해 조사 결과를 바탕으로 내용을 채워 넣습니다.

새 계획을 만들거나 기존 계획의 큰 방향을 다시 잡을 때는 반드시 Plan mode를 사용해 초안을 만든 뒤 문서로 확정하십시오. 즉, 계획 수립은 일반 구현 모드에서 바로 시작하지 말고, 먼저 Plan mode에서 범위, 수용 기준, 검증 방법, 관련 문서를 정리한 다음 ExecPlan 파일로 남겨야 합니다.

계획 문서의 위치도 단계에 따라 분명히 구분합니다. 승인 전 초안은 임시 위치에 있어도 되지만, 사용자 승인을 받은 계획은 반드시 `docs/exec-plans/active/` 아래에 둡니다. 원칙적으로는 그 계획이 PR과 merge까지 끝난 뒤에만 같은 파일을 `docs/exec-plans/completed/`로 이동합니다. 다만 사용자 지시나 상위 기준 변경으로 계획이 stale 또는 superseded 상태가 되면, 종료 이유와 대체 기준을 문서에 적은 뒤 `completed/`로 이동할 수 있습니다. 구현이 얼추 끝났다는 이유만으로 `completed/`로 옮기지 말고, merge 완료 또는 명시적 종료를 기준으로 옮겨야 합니다.

실행 가능한 명세서인 ExecPlan을 구현할 때는 사용자에게 "다음 단계"를 물어보지 말고 다음 마일스톤으로 곧바로 진행하십시오. 모든 섹션은 항상 최신 상태여야 하며, 멈추는 시점마다 목록 항목을 추가하거나 쪼개서 지금까지 끝난 일과 다음에 할 일을 명시적으로 기록해야 합니다. 모호한 부분은 스스로 해소하고, 진행 중에는 자주 커밋하십시오.

실행 가능한 명세서인 ExecPlan을 논의할 때는 그 문서 안에 의사결정 기록을 남겨야 합니다. 명세가 왜 바뀌었는지 나중에 읽는 사람이 분명히 이해할 수 있어야 합니다. ExecPlan은 살아 있는 문서이므로, 다른 자료 없이 ExecPlan만 다시 읽고도 작업을 재개할 수 있어야 합니다.

요구 사항이 까다롭거나 불확실성이 큰 설계를 조사할 때는 개념 검증용 마일스톤을 활용하십시오. 예를 들어 작은 시제품이나 토이 구현을 만들어 사용자의 제안이 실제로 가능한지 먼저 검증할 수 있습니다. 필요한 경우 라이브러리 소스를 직접 찾아 읽거나 확보해서 깊이 조사하고, 더 완전한 구현을 안내할 수 있도록 프로토타입 결과도 계획서에 포함하십시오.

## 언어 규칙

이 저장소의 모든 ExecPlan은 반드시 한국어로 작성해야 합니다. 제목, 본문, 마일스톤 설명, 진행 기록, 의사결정 기록, 검증 절차, 회고까지 모두 한국어로 적습니다. 코드, 명령어, 경로, 타입 이름처럼 원문 유지가 필요한 식별자만 원래 표기를 사용하고, 설명 문장은 한국어로 씁니다.

필수 섹션 이름도 한국어로 유지하십시오. 예를 들어 `Progress`는 `진행 현황`, `Surprises & Discoveries`는 `놀라움과 발견`, `Decision Log`는 `의사결정 기록`, `Outcomes & Retrospective`는 `결과 및 회고`로 작성합니다. 기존 영어 용어를 병기해야 할 특별한 이유가 있다면 처음 한 번만 한국어 뒤에 괄호로 덧붙이고, 이후에는 한국어 명칭만 사용하십시오.

## 요구 사항

다음 요구 사항은 협상 불가입니다.

* 모든 ExecPlan은 완전히 자기완결적이어야 합니다. 여기서 자기완결적이라는 것은 현재 문서만으로도 초보자가 필요한 지식과 지시를 모두 얻을 수 있다는 뜻입니다.
* 모든 ExecPlan은 살아 있는 문서입니다. 진행이 생기고, 새로운 사실을 발견하고, 설계 결정이 확정될 때마다 반드시 수정되어야 합니다. 어떤 시점의 수정본이든 다시 읽는 사람에게 충분해야 합니다.
* 모든 ExecPlan은 이 저장소를 전혀 모르는 초보자도 기능을 처음부터 끝까지 구현할 수 있게 만들어야 합니다.
* 모든 ExecPlan은 단순히 "정의에 맞는 코드 변경"이 아니라, 실제로 동작함을 눈으로 확인할 수 있는 결과를 만들어야 합니다.
* 전문 용어를 사용한다면 반드시 쉬운 한국어로 뜻을 설명해야 합니다. 설명하지 못할 용어라면 사용하지 마십시오.

항상 목적과 의도를 먼저 설명하십시오. 이 변경이 왜 중요한지, 사용자가 이전에는 할 수 없었던 무엇을 이제 할 수 있게 되는지, 그리고 그것이 실제로 동작함을 어떻게 확인할 수 있는지를 몇 문장으로 먼저 설명합니다. 그 다음에는 그 결과를 만들기 위해 무엇을 수정하고, 무엇을 실행하고, 무엇을 관찰해야 하는지 정확한 순서로 안내해야 합니다.

이 계획을 실행하는 에이전트는 파일 목록을 보고, 파일을 읽고, 검색하고, 프로젝트를 실행하고, 테스트를 돌릴 수 있습니다. 하지만 이전 문맥을 알지 못하며, 앞선 마일스톤에 담긴 암묵적 의도를 추론하지도 못합니다. 따라서 당신이 기대하는 전제는 모두 문서 안에 다시 적어야 합니다. 외부 블로그나 문서를 참고하라고 넘기지 마십시오. 필요한 지식은 이 계획 안에 당신의 말로 직접 녹여야 합니다. 어떤 ExecPlan이 이전 ExecPlan 위에 세워진 것이라면, 그 파일이 저장소에 커밋되어 있을 때만 경로를 명시해 참조할 수 있습니다. 커밋되어 있지 않다면 필요한 맥락을 현재 문서 안에 모두 포함해야 합니다.

## 형식 규칙

형식은 단순하지만 엄격합니다. 각 ExecPlan은 `md` 라벨이 붙은 하나의 단일 fenced code block으로만 작성해야 하며, 시작과 끝은 각각 세 개의 백틱으로 감싸야 합니다. 그 단일 코드 블록 안에는 추가적인 삼중 백틱 코드 블록을 중첩하지 마십시오. 명령어, 실행 로그, 패치, 코드 조각을 보여줘야 할 때는 같은 블록 안에서 들여쓰기로 표현합니다. 내부에 또 다른 코드 펜스를 넣으면 계획서 코드 블록이 조기에 닫힐 수 있으므로, 가독성은 들여쓰기로 해결하십시오.

단, 어떤 Markdown 파일의 내용이 오직 ExecPlan 하나뿐인 경우에는 파일 자체가 이미 계획서를 담는 그릇이므로 바깥쪽 삼중 백틱은 생략해도 됩니다.

문체는 평이한 서술형 문장을 기본으로 합니다. 특별한 이유가 없다면 목록보다 문장을 우선하십시오. 체크리스트, 표, 장황한 나열은 짧게 쓰면 의미가 흐려질 때만 제한적으로 사용합니다. 체크리스트는 `진행 현황` 섹션에서만 허용되며, 그곳에서는 반드시 사용해야 합니다. 이야기형 섹션은 문장 중심으로 유지하십시오.

제목은 Markdown의 `#`, `##` 등을 올바르게 사용하고, 각 제목 뒤에는 빈 줄을 두 줄 넣어 섹션이 분명히 구분되게 하십시오. 순서 목록과 비순서 목록도 올바른 Markdown 문법을 사용해야 합니다.

## 작성 지침

자기완결성과 쉬운 한국어가 가장 중요합니다. "데몬", "미들웨어", "RPC 게이트웨이", "필터 그래프"처럼 일상적인 한국어가 아닌 표현을 도입한다면, 그 즉시 뜻을 풀어 쓰고 이 저장소 안에서 그것이 어디에 나타나는지 함께 알려주십시오. 예를 들어 어떤 파일에 구현되어 있는지, 어떤 명령으로 실행되는지를 바로 이어서 설명해야 합니다. "앞에서 정의했듯이" 또는 "아키텍처 문서에 따르면" 같은 표현으로 설명을 생략하지 마십시오. 반복이 있더라도 필요한 설명은 이 문서 안에 직접 넣어야 합니다.

흔한 실패 방식도 피해야 합니다. 정의되지 않은 전문 용어에 기대지 말고, 기능의 겉모양만 맞춘 채 실제로는 의미 있는 동작이 없는 계획을 쓰지 마십시오. 중요한 결정을 독자에게 떠넘기지 마십시오. 애매한 지점이 있다면 계획서 안에서 직접 방향을 정하고, 왜 그렇게 정했는지 설명하십시오. 사용자 눈에 보이는 효과는 충분히 자세히 설명하되, 사소한 구현 디테일은 꼭 필요한 만큼만 구체화하는 편이 낫습니다.

계획은 반드시 관찰 가능한 결과에 닻을 내려야 합니다. 구현 후 사용자가 무엇을 할 수 있는지, 어떤 명령을 실행해야 하는지, 어떤 출력이 보여야 하는지를 명시하십시오. 수용 기준은 사람이 직접 검증할 수 있는 행동으로 써야 합니다. 예를 들어 "서버를 시작한 뒤 [http://localhost:8080/health](http://localhost:8080/health)에 접속하면 HTTP 200과 본문 `OK`가 반환된다"처럼 써야지, "HealthCheck 구조체를 추가한다"처럼 내부 속성만 적어서는 안 됩니다. 변경이 내부 구현에만 머물더라도 그 영향이 어떻게 검증되는지 설명해야 합니다. 예를 들면, 변경 전에는 실패하고 변경 후에는 성공하는 테스트와 실제 사용 시나리오를 함께 제시할 수 있습니다.

저장소 맥락도 명시적으로 적으십시오. 파일은 저장소 루트 기준 경로를 정확히 적고, 함수와 모듈 이름도 구체적으로 쓰십시오. 새 파일을 만든다면 어느 경로에 만들어야 하는지 분명히 적어야 합니다. 여러 영역을 함께 건드린다면, 초보자가 길을 잃지 않도록 각 부분이 어떻게 연결되는지 짧은 방향 설명을 먼저 덧붙이십시오. 명령어를 제시할 때는 작업 디렉터리와 정확한 커맨드 라인을 함께 적으십시오. 환경에 따라 결과가 달라질 수 있다면 전제 조건을 적고, 합리적이라면 대안도 제시하십시오.

계획은 반복 실행해도 안전해야 합니다. 같은 단계를 여러 번 수행해도 환경이 깨지지 않도록 절차를 쓰십시오. 중간에 실패할 수 있는 단계라면 어떻게 다시 시도할지, 어디서부터 복구할지를 안내해야 합니다. 마이그레이션이나 파괴적 작업이 필요하다면 백업 방법이나 안전한 대안을 구체적으로 적으십시오. 가능하면 추가적이고 검증 가능한 변경을 선호하고, 각 단계마다 결과를 확인할 수 있게 구성하십시오.

검증은 선택 사항이 아닙니다. 테스트 실행 방법, 필요하다면 시스템을 띄우는 방법, 그리고 실제로 유용한 동작을 관찰하는 방법을 반드시 포함해야 합니다. 새 기능이나 새 능력이 생겼다면 충분한 테스트 범위를 설명해야 합니다. 예상 출력과 오류 메시지도 적어 두어 초보자가 성공과 실패를 구분할 수 있게 해야 합니다. 가능하다면 단순 컴파일 성공을 넘어 변경이 실제로 효과가 있음을 증명하는 절차를 포함하십시오. 예를 들어 작은 엔드 투 엔드 시나리오, CLI 예제, HTTP 요청/응답 예시가 이에 해당합니다. 프로젝트 도구 체인에 맞는 정확한 테스트 명령을 적고, 그 결과를 어떻게 해석해야 하는지도 설명하십시오.

증거도 남겨야 합니다. 명령 수행 결과로 터미널 출력, 짧은 diff, 로그가 생긴다면 단일 코드 블록 안에 들여쓴 예시로 포함하십시오. 길게 붙여 넣기보다, 성공을 입증하는 핵심만 짧고 선명하게 제시해야 합니다. 패치를 보여줘야 한다면 대형 블롭 전체를 넣기보다 파일 단위 diff나 작은 발췌문처럼 독자가 계획을 따라가며 스스로 재현할 수 있는 형태를 선호하십시오.

## 마일스톤

마일스톤은 형식적인 관리 항목이 아니라 작업의 서사여야 합니다. 마일스톤으로 일을 나눈다면, 각 마일스톤은 짧은 문단으로 시작해야 합니다. 그 문단에는 이번 범위가 무엇인지, 끝났을 때 이전에는 없던 어떤 결과가 생기는지, 어떤 명령을 실행할 것인지, 어떤 수용 결과를 기대하는지가 모두 들어 있어야 합니다. 읽는 사람이 자연스럽게 따라갈 수 있도록 목표, 작업, 결과, 검증의 흐름으로 쓰십시오.

`진행 현황`과 마일스톤은 서로 다른 역할을 합니다. 마일스톤은 큰 흐름과 이야기를 설명하고, `진행 현황`은 세부 작업 상태를 추적합니다. 두 섹션 모두 반드시 있어야 합니다. 짧게 쓰기 위해 핵심 정보를 생략하지 마십시오. 미래의 구현자가 실제로 필요로 할 만한 세부 사항은 빠뜨리면 안 됩니다.

각 마일스톤은 그 자체로 검증 가능해야 하며, 전체 목표를 향해 점진적으로 의미 있는 결과를 만들어야 합니다.

## 살아 있는 계획서와 설계 결정

* ExecPlan은 살아 있는 문서입니다. 중요한 설계 결정을 내릴 때마다 결정 내용뿐 아니라 그렇게 판단한 이유도 함께 업데이트하십시오. 모든 결정은 `의사결정 기록` 섹션에 남겨야 합니다.
* 모든 ExecPlan에는 `진행 현황`, `놀라움과 발견`, `의사결정 기록`, `결과 및 회고` 섹션이 반드시 있어야 하며, 작업 내내 최신 상태로 유지되어야 합니다.
* 최적화 동작, 성능 상충 관계, 예상치 못한 버그, 되돌리기 어려운 동작 방식처럼 접근 방식을 바꾼 발견이 있다면 `놀라움과 발견` 섹션에 짧은 증거와 함께 기록하십시오. 가능하면 테스트 출력처럼 재현 가능한 증거를 붙이십시오.
* 구현 중간에 방향을 바꿨다면 그 이유를 `의사결정 기록`에 남기고, 그에 따른 영향은 `진행 현황`에도 반영해야 합니다. 계획서는 현재 작업자만 위한 체크리스트가 아니라, 다음 기여자를 위한 안내서이기도 합니다.
* 큰 작업을 마쳤거나 전체 계획을 완료했을 때는 `결과 및 회고` 항목을 작성해 무엇을 달성했고 무엇이 남았는지, 어떤 교훈이 있었는지 요약하십시오.

## 프로토타이핑 마일스톤과 병렬 구현

더 큰 변경의 위험을 줄일 수 있다면, 명시적인 프로토타이핑 마일스톤을 포함해도 좋고 오히려 권장되기도 합니다. 예를 들어 의존 라이브러리의 저수준 연산자를 먼저 붙여 실현 가능성을 검증하거나, 두 가지 조합 순서를 따로 시도해 최적화 차이를 측정하는 식입니다. 이런 프로토타입은 추가적이고 검증 가능해야 합니다. 범위를 분명히 "프로토타이핑"이라고 표시하고, 어떻게 실행하고 어떤 결과를 관찰해야 하는지, 그리고 어떤 기준으로 채택하거나 폐기할지를 적으십시오.

위험을 줄이기 위해서는 먼저 추가적인 코드 변경을 넣고, 이후 테스트를 계속 통과시키면서 점진적으로 오래된 경로를 제거하는 방식이 좋습니다. 예를 들어 마이그레이션 중에는 새 어댑터와 기존 경로를 잠시 병렬로 유지할 수 있습니다. 이런 선택을 했다면 두 경로를 어떻게 검증할지, 나중에 하나를 어떻게 안전하게 제거할지 테스트 기준과 함께 설명하십시오. 여러 새 라이브러리나 기능 영역을 동시에 다룬다면, 각각을 서로 독립적으로 검증하는 스파이크를 두는 것도 좋습니다. 외부 라이브러리가 기대한 기능을 실제로 제공하는지, 필요한 기능이 격리된 조건에서 동작하는지를 먼저 증명하십시오.

## 좋은 ExecPlan의 골격

    # <짧고 행동 지향적인 제목>

    이 ExecPlan은 살아 있는 문서입니다. `진행 현황`, `놀라움과 발견`, `의사결정 기록`, `결과 및 회고` 섹션은 작업이 진행되는 내내 최신 상태로 유지해야 합니다.

    저장소에 `PLANS.md`가 커밋되어 있다면, 저장소 루트 기준 경로를 이 문서에 적고 이 계획서가 `PLANS.md`를 준수하며 관리되어야 한다는 점을 명시합니다.

    ## 목적 / 큰 그림

    이 변경 이후 사용자가 얻게 되는 가치와 그것이 어떻게 동작하는지 확인하는 방법을 몇 문장으로 설명합니다. 사용자 눈에 보이는 동작을 분명히 적습니다.

    ## 진행 현황

    체크박스 목록으로 세부 단계를 요약합니다. 작업을 멈추는 모든 시점은 여기에 반영되어야 하며, 필요하다면 하나의 항목을 "완료된 부분"과 "남은 부분"으로 쪼개야 합니다. 이 섹션은 항상 실제 현재 상태와 일치해야 합니다.

    - [x] (2025-10-01 13:00Z) 예시 완료 항목
    - [ ] 예시 미완료 항목
    - [ ] 예시 부분 완료 항목 (완료: X, 남음: Y)

    진행 속도를 판단할 수 있도록 타임스탬프를 사용합니다.

    ## 놀라움과 발견

    구현 중 발견한 예상 밖의 동작, 버그, 최적화 포인트, 통찰을 기록합니다. 짧은 증거를 함께 제시합니다.

    - 관찰:
      증거:

    ## 의사결정 기록

    작업 중 내린 모든 결정을 아래 형식으로 기록합니다.

    - 결정:
      근거:
      날짜/작성자:

    ## 결과 및 회고

    주요 마일스톤 또는 전체 완료 시점에 결과, 남은 간극, 배운 점을 요약합니다. 처음의 목적과 비교해 설명합니다.

    ## 맥락과 길잡이

    독자가 아무것도 모른다고 가정하고 현재 상태를 설명합니다. 핵심 파일과 모듈은 전체 경로로 적습니다. 익숙하지 않은 용어는 모두 정의합니다. 이전 계획서를 알고 있다고 가정하지 않습니다.

    ## 작업 계획

    어떤 수정과 추가를 어떤 순서로 할지 문장으로 설명합니다. 각 수정마다 파일 경로, 위치(함수, 모듈 등), 무엇을 넣거나 바꿀지를 구체적으로 적습니다. 불필요하게 장황하지 않되, 실행에는 부족함이 없어야 합니다.

    ## 구체적인 단계

    어디에서 어떤 명령을 실행할지 정확히 적습니다. 명령이 출력을 만든다면 짧은 예상 예시도 함께 보여 줍니다. 이 섹션은 작업이 진행될수록 함께 갱신되어야 합니다.

    ## 검증과 수용 기준

    시스템을 어떻게 시작하거나 실행해 볼지, 그리고 무엇을 관찰해야 하는지 설명합니다. 수용 기준은 구체적인 입력과 출력이 있는 동작으로 씁니다. 테스트가 포함된다면 "프로젝트의 테스트 명령을 실행했을 때 어떤 테스트가 변경 전에는 실패하고 변경 후에는 통과하는지"까지 적습니다.

    ## 반복 실행 가능성 및 복구

    같은 단계를 반복해도 안전하다면 그렇게 명시합니다. 위험한 단계가 있다면 안전한 재시도나 롤백 방법을 제공합니다. 완료 후에는 환경이 깔끔하게 남아야 합니다.

    ## 산출물과 메모

    성공을 입증하는 데 가장 중요한 실행 로그, diff, 코드 발췌를 들여쓴 예시로 포함합니다. 짧고 핵심적으로 유지합니다.

    ## 인터페이스와 의존성

    사용할 라이브러리, 모듈, 서비스와 그 이유를 단정적으로 적습니다. 마일스톤이 끝났을 때 존재해야 하는 타입, 인터페이스, 함수 시그니처도 명시합니다. 경로와 이름은 `crate::module::function` 또는 `package.submodule.Interface`처럼 안정적인 형식으로 적습니다. 예:

        pub trait Planner {
            fn plan(&self, observed: &Observed) -> Vec<Action>;
        }

위 지침을 따르면 문맥이 없는 단일 에이전트나 저장소를 처음 보는 사람도 ExecPlan을 처음부터 끝까지 읽고 실제로 동작하는 결과를 만들 수 있어야 합니다. 기준은 명확합니다. 자기완결적이어야 하고, 스스로 충분해야 하며, 초보자를 안내할 수 있어야 하고, 결과 중심이어야 합니다.

계획서를 수정할 때는 변경 내용이 살아 있는 문서의 모든 섹션에 일관되게 반영되었는지 반드시 확인해야 합니다. 그리고 계획서 맨 아래에는 무엇을 왜 바꿨는지 설명하는 변경 메모를 남겨야 합니다. ExecPlan은 무엇을 할지뿐 아니라, 왜 그렇게 해야 하는지도 거의 모든 중요한 지점에서 설명해야 합니다.
