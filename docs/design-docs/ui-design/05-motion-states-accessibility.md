# Motion States Accessibility

## 목적

이 문서는 모션, 상태 표현, 접근성 규칙을 함께 다룬다.

## 모션

이 프로젝트는 모션을 금지하지 않는다.
다만 목적 없는 애니메이션은 허용하지 않는다.

허용:

- hover 색상 변화
- 1~2% 수준의 미세 확대
- 모달/오버레이 진입
- 배경 별, 숫자 카운팅, 차트 보조 애니메이션

주의:

- 데이터 읽는 중 시선을 빼앗는 반복 애니메이션은 최소화한다.
- 한 화면에 여러 모션 스타일을 과하게 섞지 않는다.
- 장식 모션은 상태 전달 모션보다 약해야 한다.

페이지 진입:

- 로딩 화면 뒤 실제 페이지는 일반 화면 기준 300~440ms 안에서 눈에 보이게 등장시킨다.
- 기본 진입은 `opacity + y/scale + 약한 blur` 조합을 사용하고, 도메인별 변주는 작게 둔다.
- 트레이딩 상세처럼 실시간 데이터를 읽는 화면은 160~220ms 안에서 빠르고 선명하게 유지한다.
- 마켓 목록/마이페이지/관심/커뮤니티/상점/회원가입은 트레이딩보다 진입감을 더 분명하게 표현할 수 있다.
- `prefers-reduced-motion` 환경에서는 위치/확대 이동 없이 즉시 표시한다.

## 상태 표현

새 UI는 정상 상태만 설계하면 안 된다.

항상 고려할 상태:

- loading
- empty
- error
- unauthenticated
- 설정 미완료

현재 서비스에서 좋은 패턴:

- 스켈레톤 로딩
- 블러 처리된 비활성 콘텐츠 + 안내 문구
- 카드형 에러 표현
- toast 기반 즉시 피드백

규칙:

- 페이지 전체 실패보다 카드 단위 실패 복구를 우선한다.
- 빈 상태는 설명이 있는 상태로 보여 준다.
- 에러 표현은 빨간 계열이되 과도하게 위협적이지 않게 한다.
- toast는 짧고 즉각적인 결과 알림에만 사용한다.

## 접근성

현재 서비스는 데스크톱 우선이지만 접근성 기준은 낮추지 않는다.

- 클릭 가능한 영역은 충분한 크기를 가진다.
- hover만 있고 focus가 없는 컴포넌트를 늘리지 않는다.
- 아이콘 버튼은 `aria-label`을 가져야 한다.
- 색상만으로 등락이나 선택 상태를 전달하지 않는다.
- 검색, 모달, 드롭다운은 키보드 이동 흐름을 고려한다.

## 체크리스트

- 모션이 정보 읽기를 방해하지 않는가
- loading, empty, error 상태가 모두 설계되었는가
- 색만으로 의미를 전달하고 있지 않은가
- 키보드만으로 주요 플로우 접근이 가능한가
