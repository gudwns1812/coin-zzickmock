# Backend AGENTS Guide

이 문서는 `coin-zzickmock` 백엔드 작업의 단일 운영 규칙이다.
백엔드 코드 변경(기능 추가, 리팩토링, 버그 수정, 테스트 수정) 전에 반드시 이 문서를 기준으로 설계와 구현 방향을 정한다.

## 목표

1. 기능은 유지하고 구조를 개선한다.
2. 책임이 드러나는 객체지향 구조를 유지한다.
3. 예외/응답/패키지 구조를 일관되게 유지한다.
4. 회귀 테스트로 기존 동작을 보호한다.

## 작업 시작 전 필수 절차

1. 변경 범위를 `core`, `storage`, `support`, `extern`, `batch` 중 어디에 두어야 하는지 먼저 판단한다.
2. Service에 유스케이스 흐름 외 세부 로직이 들어가려는지 확인한다.
3. 응답은 `ApiResponse`, 예외는 `CoreException` + `ErrorType`으로 일관되게 처리할 계획인지 확인한다.
4. 구조 변경 시 회귀 테스트와 필요한 단위 테스트 계획을 먼저 잡는다.

## 최상위 패키지 규칙

- `core`: 도메인과 유스케이스
- `storage`: 저장소/영속화 구현
- `support`: 도메인 비종속 공통 지원 기능
- `extern`: 외부 시스템 연동
- `batch`: 스케줄링/정기 실행 작업

원칙:

- 패키지는 "파일 수"가 아니라 "책임 경계"로 나눈다.
- 기존 패키지 어디에도 자연스럽게 들어가지 않으면, 새 최상위 패키지 추가를 허용한다.
- `global` 같은 포괄적 이름은 지양하고, 이름만으로 역할이 드러나야 한다.

## 계층 구조 및 의존 규칙

백엔드는 다음과 같은 논리 계층으로 본다.

1. 표현 계층: `core.api
2. 비즈니스 계층: `core.application`
3. 구현 계층: `core.application내 implement`
4. 도메인 계층: `core.domain`

계층 규칙:

- 상위 계층에서 하위 계층으로 의존성 방향이 흘러야한다. 역방향 참조 금지.
- 상위 계층에서 하위 계층으로 2계층이상을 넘는 의존은 금지한다.
- 같은 계층에서 서로 참조하는것 또한 금지한다.
- 다만, 구현계층에서만 서로 참조하는 것을 허용한다. 이는 재사용성을 높이기 위함이다.
  예를 들어 Processor가 finder나 Reader를 의존하는것은 재사용을 높이는 방향이기 때문에 허용한다.
- core를 제외한 나머지는 계층 규칙은 적용하지 않거나 적용하더라도 엄격하게 사용하지 않는다.

판단 기준:

- Service에서 흐름이 보여야 하고, 구현 세부 사항은 하위 협력 객체로 내려가야 한다.
- 구현 계층은 재사용을 위해 분리할 수 있지만, 한 메서드당 한 클래스 식의 과도한 분할은 지양한다.
- 구현 계층 재사용은 "같은 계층 내 재사용"으로 해결하고, 역방향 의존이나 우회 의존으로 해결하지 않는다.

## 계층별 규칙

### core

- `api`: Controller, Request DTO, Response DTO
- `application`: Service, 유스케이스 조합, 구현 협력 객체
- `domain`: 도메인 상태와 도메인 행위

### storage

- 저장 기술부터 분리한다 (`db`, `redis`).
- 그 아래는 도메인/책임 단위로 세분화한다.
- 예: `storage.db.member.entity`, `storage.db.member.repository`, `storage.redis.stock.publisher`
- Repository는 조회/저장에 집중한다.
- db 조회할때는 성능 이슈에 집중한다. N+1문제나, 풀테이블 스캔이 발생하지는 않도록 한다.(함수를 이용한 인덱스 무력화 금지)

### support

- 인증/인가, 공통 응답, 공통 예외, 공통 설정만 둔다.
- 외부 API 연동, 배치, 특정 도메인 업무 로직은 두지 않는다.

### extern

- 외부 API/SDK/프로토콜 적응 책임은 `extern`에 둔다.
- 외부 예외는 CoreException을 사용하지 않는다. (Core가 아니기 때문)
- 외부 연동 객체는 유스케이스 전체를 수행하지 않고 통신 결과 반환까지만 담당한다.

### batch

- `@Scheduled`는 `batch`에만 둔다.
- 배치 클래스는 실행 시점과 흐름 오케스트레이션만 담당한다.
- 외부 호출/저장/발행 세부 구현은 협력 객체로 위임한다.

## Service 및 협력 객체 규칙

- Service는 유스케이스 진입점/트랜잭션 경계/협력 객체 조합/최종 DTO 반환만 담당한다.
- 아래 책임은 Service에 직접 두지 않는다.
    - DB/Redis 조회 세부 구현
    - 외부 API 호출 세부 구현
    - 검색 분기 규칙
    - 정렬/페이지 계산
    - 응답 DTO 조립 세부 로직
    - 복잡 계산/판단 규칙
- Service의 private 메서드가 비대해지면 협력 객체 분리 신호로 본다.

## implement 계층 기본 원칙:

- 클래스 남발보다 "도메인 단위 클래스 통합 + 메서드 분리"를 우선한다.
- 클래스 분리는 변경 이유/의존성 경계/테스트 격리 이점이 명확할 때만 한다.

## Domain/DTO 규칙

- Domain은 상태와 도메인 행위를 가진다.
- 의미 있는 상태 변경은 엔티티 메서드로 둔다.
- 표현 DTO 조립 책임은 엔티티에 두지 않는다.
- 도메인 변환은 DTO/엔티티 내부 `toDomain()`으로 처리한다.

DTO 규칙:

- 별도 매퍼 계층 대신 DTO 정적 팩토리(`from`, `of`)를 우선한다.
- Service에서 DTO를 필드 단위로 수동 조립하지 않는다.
- Request DTO는 입력 검증/전달 책임만 가진다.
- Response DTO는 표현 책임만 가진다.

## 예외/응답/컨트롤러 규칙

예외:

- 비즈니스 예외는 `CoreException`으로 통일한다.
- `ErrorType`은 `ErrorCode`, 메시지, `LogLevel`, `HttpStatus`를 가진다.
- 컨트롤러에서 실패 응답을 직접 조립하거나 상태코드 분기를 직접 만들지 않는다.

응답:

- 응답은 `ApiResponse` 하나로 통일한다.
    - 성공: `ApiResponse.success()`
    - 성공 데이터: `ApiResponse.success(data)`
    - 실패: `ApiResponse.error(data, errorType)`
- 응답 필드 구조(`result`, `data`, `error`)를 유지한다.

컨트롤러:

- HTTP 입출력과 Service 호출만 담당한다.
- 비즈니스 규칙 판단, 조회 전략 분기, DTO 상세 조립을 넣지 않는다.
- 상태코드는 `HttpStatus` enum 사용을 원칙으로 한다.
- 단일 상태코드면 `@ResponseStatus`, 다중 상태코드면 `ResponseEntity`를 사용한다.

## 네이밍 규칙

- 이름은 역할을 설명해야 한다.
- 축약보다 의미 전달을 우선한다.
- `global`, `util`, `commonService`, `Helper`, `Manager` 같은 포괄명은 지양한다.
- Core에서는 특정 기술에 종속되는 이름은 지양한다.

## 리팩토링 기준

아래 항목이 보이면 리팩토링 대상으로 본다.

- 계층 책임 혼합
- 일관성 붕괴
- 규칙의 중복 분산
- 비대해진 Service
- Service에 퍼진 DTO 변환
- 컨트롤러의 예외/상태코드 직접 조립
- `support`/도메인 계층에 섞인 외부 연동
- 업무 로직과 뒤섞인 배치 로직

## 테스트 규칙

- 리팩토링 전후 비즈니스 동작은 동일해야 한다.
- 구조 변경 시 회귀 테스트를 유지/보강한다.
- 분리한 규칙 객체의 단위 테스트를 추가한다.
- 변경 모듈 테스트 우선, 대규모 구조 변경이나 마무리 단계에서 전체 테스트

권장 우선순위:

1. 기존 API 회귀 테스트
2. 새 협력 객체/규칙 객체 단위 테스트
3. 예외 응답/상태코드 테스트
4. 외부 연동 파싱/fallback 테스트

## 리뷰 체크리스트

리뷰 시 아래를 우선 확인한다.

1. 현재 패키지 배치 이유가 명확한가
2. Service가 얇게 유지되는가
3. 예외/응답 규칙이 지켜지는가
4. 이름만 보고 책임 추론이 가능한가
5. 일관성이 깨진 지점이 없는가
6. 테스트가 구조 변경을 보호하는가

## 최종 원칙

빠른 구현보다 장기 유지보수 가능한 구조를 우선한다.
기능 추가 전 구조를 먼저 맞추고, 일관성이 깨진 부분은 범위 안에서 함께 바로잡는다.
