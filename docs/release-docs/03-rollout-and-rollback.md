# 03. Rollout And Rollback

## Purpose

이 문서는 `coin-zzickmock`의 롤아웃 순서, 호환성 원칙, 롤백과 장애 대응 기준을 정의한다.
핵심 목표는 "문제가 생겼을 때 당황하지 않고 같은 기준으로 멈추고 되돌리는 것"이다.

## Rollout Principles

- 사용자 영향이 큰 환경일수록 작은 범위부터 확인한다.
- 프론트와 백엔드가 함께 바뀌면 호환성 방향을 먼저 확인한다.
- 한 번에 여러 위험 요소를 동시에 바꾸지 않는다.
- 스키마, 설정값, 코드 변경을 묶어야 할 때는 순서를 문서에 먼저 적는다.

## Safe Order For Full-stack Changes

### Backward-compatible Change

백엔드가 새 필드나 새 API를 추가하는 식의 하위 호환 변경이라면 아래 순서를 권장한다.

1. 백엔드 배포
2. 백엔드 스모크 테스트
3. 프론트 배포
4. 프론트 스모크 테스트

### Potentially Breaking Change

기존 응답, 스키마, 인증 흐름을 깨뜨릴 수 있는 변경은 아래 중 하나를 반드시 사용한다.

- 두 단계 배포
- 기능 플래그
- expand and contract 방식

즉, "백엔드와 프론트를 동시에 올리면 되겠지" 방식으로 처리하지 않는다.

## Database Change Rule

DB 변경은 아래 원칙을 따른다.

- 가능하면 먼저 확장한다.
- 새 코드가 옛 스키마와 잠시 공존할 수 있게 설계한다.
- 데이터 이전이 필요하면 별도 단계로 분리한다.
- 최종 제거는 안정화 이후 후속 릴리즈에서 처리한다.

스키마 변경이 있으면 코드만이 아니라 [docs/generated/db-schema.md](/Users/hj.park/projects/coin-zzickmock/docs/generated/db-schema.md)와 관련 운영 기록도 함께 갱신한다.

## Rollback Trigger

아래 중 하나라도 발생하면 롤백 또는 긴급 차단을 우선 검토한다.

- 서비스 기동 실패
- 로그인 또는 핵심 진입 흐름 불가
- 주요 API 5xx 급증
- 데이터 손상 또는 중복 처리 징후
- 보안 정책 위반 또는 민감 정보 노출 가능성
- 프론트에서 치명적 런타임 에러가 다수 발생

## Rollback Rules

- 롤백 기준점은 배포 전에 확정한다.
- 롤백은 새 변경을 덮는 임시 수정보다 우선 검토한다.
- 데이터 마이그레이션이 되돌릴 수 없는 경우에는 서비스 보호 조치를 먼저 수행한다.
- 롤백 후에도 원인과 영향 범위를 기록한다.

## Incident Response Flow

1. 장애 징후를 확인한다.
2. 신규 릴리즈와 연관된 증상인지 판단한다.
3. 사용자 영향과 확산 범위를 파악한다.
4. 롤백, 기능 차단, 핫픽스 중 우선 조치를 고른다.
5. 조치 이후 스모크 테스트를 다시 수행한다.
6. 운영 기록에 원인 가설과 다음 액션을 남긴다.

## Ownership

- 릴리즈 실행자는 배포 성공만이 아니라 사후 확인까지 책임진다.
- 핫픽스가 필요하면 기존 릴리즈 기록과 연결한다.
- 롤백이 발생하면 다음 릴리즈 전에 재발 방지 항목을 문서화한다.

## Maintenance Rule

- 실제 인프라가 정해지면 환경별 구체 명령과 대시보드 위치를 이 문서에 연결한다.
- 롤백 경험이 생기면 사례 기반 체크 항목을 이 문서에 추가한다.
