# 백엔드 개발 가이드라인

## 목적

이 문서는 이 프로젝트의 백엔드 개발 규율이다.
앞으로 기능을 추가하거나 리팩토링할 때, 구현자와 리뷰어는 이 문서를 기준으로 일관성을 판단한다.

핵심 목표는 아래 4가지다.

1. 기능은 유지하고 구조만 개선한다.
2. 책임이 드러나는 객체지향 구조를 유지한다.
3. 예외, 응답, 패키지 구조를 통일한다.
4. 회귀 테스트로 기존 동작을 반드시 보호한다.

## 최상위 패키지 규칙

현재 자주 사용하는 최상위 패키지는 아래와 같다.
하지만 이 목록을 고정 규칙으로 취급하지 않는다.
새로운 책임이 분명하다면, 그 책임을 설명할 수 있는 새 최상위 패키지를 추가할 수 있다.

- `core`: 도메인과 유스케이스
- `storage`: 저장소와 영속화 구현
- `support`: 공통 지원 기능
- `extern`: 외부 시스템 연동
- `batch`: 스케줄링, 정기 실행 작업

중요한 것은 개수가 아니라 책임의 경계다.
최상위 패키지는 역할이 명확하게 분리될 때만 추가하고, 단순히 파일이 많아졌다는 이유로 만들지 않는다.

### `core`

`core`는 이 서비스의 본질적인 업무를 담는다.

- `api`: Controller, Request DTO, Response DTO
- `application`: Service, 유스케이스 조합, 구현 협력 객체
- `domain`: Entity, 도메인 상태, 도메인 행위

### `storage`

`storage`는 저장 기술을 드러내는 계층이다.

- `db`: JPA Repository, DB 엔티티 저장 구현
- `redis`: Redis DTO, publisher, subscriber, redis 접근 구현

### `support`

`support`는 도메인이 아닌 공통 지원 기능만 둔다.

- 인증과 인가
- 공통 응답
- 공통 예외
- 공통 설정

아래 항목은 `support`에 두지 않는다.

- 외부 API 연동
- 스케줄러
- 배치 작업
- 특정 도메인 전용 업무 로직

### `extern`

외부 API, 외부 SDK, 외부 시스템 프로토콜 적응은 모두 `extern`에 둔다.

- 외부 호출 클라이언트
- 외부 응답 DTO
- 외부 시스템 전용 변환

### `batch`

정기 실행 작업은 모두 `batch`에 둔다.

- `@Scheduled` 사용 클래스
- 배치 시작점
- 배치 오케스트레이션

배치는 외부 연동이나 저장 구현을 직접 모두 품지 않는다.
필요한 협력 객체를 주입받아 호출만 한다.

## 패키지 배치 원칙

패키지는 기술명보다 책임이 먼저 보이게 배치한다.

- 도메인 로직이면 `core`
- 저장 관련 책임이면 `storage`
- 외부 시스템 연동이면 `extern`
- 배치 실행이면 `batch`
- 공통 지원이면 `support`

`global` 같은 포괄적인 이름은 사용하지 않는다.
패키지명만 보고 역할을 추측할 수 있어야 한다.
현재 존재하는 최상위 패키지 목록에 억지로 맞추지 않는다.
기존 패키지 어디에도 자연스럽게 들어가지 않는다면, 먼저 새로운 책임인지 판단하고 필요하면 새 상위 패키지를 만든다.

## Service 규칙

Service는 유스케이스 흐름만 보여주는 얇은 계층이어야 한다.

Service가 가져야 하는 책임은 아래까지만 허용한다.

- 유스케이스 진입점
- 트랜잭션 경계
- 필요한 협력 객체 조합
- 최종 DTO 반환

Service가 직접 가지면 안 되는 책임은 아래와 같다.

- Redis/DB 조회 세부 구현
- 외부 API 호출 세부 구현
- 검색 분기 규칙
- 정렬과 페이지 계산
- 응답 DTO 조립 세부 로직
- 복잡한 계산 규칙

Service 안에 private 메서드가 계속 늘어나기 시작하면 책임 분리 신호로 본다.

## 구현 협력 객체 규칙

`core.application.<domain>.implement`는 Service에서 분리한 세부 책임을 담는다.
다만 `Reader`, `Writer`, `Processor` 같은 이름은 예시일 뿐이며 고정 규칙이 아니다.

이 계층의 네이밍은 책임이 가장 잘 드러나는 방향으로 정한다.

예시:

- 조회면 `Loader`, `Reader`
- 저장이면 `Saver`, `Writer`
- 계산이면 `Calculator`
- 규칙 판단이면 `Policy`
- 조합이면 `Handler`, `Executor`

중요한 것은 이름이 아니라 아래 기준이다.

- 하나의 클래스는 하나의 이유로만 변경되어야 한다.
- 하나의 클래스가 조회, 저장, 계산, 변환을 동시에 가지지 않는다.
- Service가 다시 세부 구현을 품지 않도록 협력 객체로 분리한다.

## Domain 규칙

Domain은 상태와 도메인 행위를 가진다.

- 의미 있는 상태 변경은 엔티티 메서드로 둔다.
- 단순 응답 조립은 엔티티에 두지 않는다.
- 표현 계층용 DTO 생성 책임은 엔티티에 두지 않는다.

도메인으로 변환은 DTO 혹은 엔티티 내부에서 toDomain()을 사용해 변환한다.

## DTO 규칙

DTO 변환은 별도 매퍼 계층 대신 DTO 정적 팩토리로 통일한다.

허용:

- `SearchResponseDto.from(stock)`
- `CategoryPageResponseDto.of(result)`
- `StockDto.from(stock)`

지양:

- Service에서 DTO 필드를 일일이 꺼내서 `new` 하는 방식
- DTO 내부에 업무 계산 로직을 넣는 방식

DTO는 아래 규칙을 따른다.

- Request DTO는 입력 검증과 전달 역할만 가진다.
- Response DTO는 표현 역할만 가진다.
- 계산이 필요하면 DTO 밖에서 끝낸 뒤, DTO는 결과를 받기만 한다.

## 예외 규칙

비즈니스 예외는 `CoreException` 하나로 통일한다.

- `CoreException`은 `ErrorType`을 가진다.
- `ErrorType`은 `ErrorCode`, 메시지, `LogLevel`, `HttpStatus`를 가진다.
- `ExceptionHandler`는 `ErrorType` 기준으로 응답과 로그를 처리한다.

예외 규칙은 아래와 같다.

- 비즈니스 실패는 예외로 표현한다.
- 컨트롤러에서 실패 응답을 직접 조립하지 않는다.
- 상태코드 분기를 컨트롤러에서 직접 만들지 않는다.
- 중복, 권한 없음, 미존재 같은 실패도 예외 흐름으로 통일한다.

## 응답 규칙

응답은 `ApiResponse` 하나로 통일한다.

- 성공: `ApiResponse.success()`
- 성공 데이터 포함: `ApiResponse.success(data)`
- 실패: `ApiResponse.error(data, errorType)`

응답 필드는 아래 구조를 유지한다.

- `result`
- `data`
- `error`

`SuccessResponse`, `ErrorResponse`처럼 별도 래퍼를 늘리지 않는다.

## Controller 규칙

Controller는 HTTP 입출력만 담당한다.

- 요청 파라미터 수신
- 인증 주체 수신
- Service 호출
- `ApiResponse` 반환

Controller에서 하지 말아야 할 것:

- 비즈니스 규칙 판단
- 조회 전략 분기
- 예외 응답 조립
- DTO 상세 조립

상태코드 규칙은 아래와 같다.

- 하나의 상태코드만 반환하면 `@ResponseStatus(HttpStatus.X)` 사용
- 여러 상태코드가 필요하면 `ResponseEntity` 사용
- 숫자 상수나 별도 status 상수는 금지
- 반드시 `HttpStatus.OK`, `HttpStatus.CREATED` 같은 enum 사용

## 외부 연동 규칙

외부 API 연동은 `extern`에 둔다.

- 외부 URL, 프로토콜, 파싱 책임은 클라이언트가 가진다.
- 외부 응답 DTO도 `extern` 하위에 둔다.
- 외부 장애나 파싱 실패는 `CoreException`으로 변환한다.

외부 연동 객체는 도메인 유스케이스를 직접 수행하지 않는다.
외부 시스템과 통신하고 결과를 반환하는 역할까지만 가진다.

## 배치 규칙

`@Scheduled`는 `batch`에만 둔다.

- 배치 클래스는 실행 시점과 실행 흐름만 가진다.
- 외부 호출, 저장, 발행 세부 구현은 협력 객체로 위임한다.
- 배치 클래스명은 역할이 드러나야 한다.

지양:

- `logWarm` 같은 의미가 약한 이름
- 배치 클래스가 외부 호출 파싱부터 저장까지 전부 수행하는 구조

## 저장 계층 규칙

저장 계층은 최상위에서 기술을 나누고, 그 아래에서 도메인 또는 책임 단위로 나눈다.

예시:

- `storage.db.member.entity`
- `storage.db.member.repository`
- `storage.db.member.config`
- `storage.redis.stock.dto`
- `storage.redis.stock.publisher`

즉, `storage -> 기술 -> 도메인/책임 -> 세부구현` 순서로 구조를 잡는다.
`db`, `redis` 아래에 모든 도메인 파일을 평평하게 두지 않는다.

저장 계층 구조를 잡을 때는 아래 기준을 따른다.

- 어떤 저장 기술을 사용하는지 먼저 드러나야 한다.
- 같은 도메인의 저장 관련 요소는 가까이 모여 있어야 한다.
- `entity`, `repository`, `config`, `publisher`, `subscriber`, `dto` 같은 세부 책임은 도메인 하위에서 구분한다.
- 저장 계층 구조도 일관성이 깨지면 리팩토링 대상으로 본다.

Repository는 조회/저장에 집중한다.
정렬 규칙, fallback 정책, 응답 조립은 Repository에 넣지 않는다.

## 네이밍 규칙

이름은 역할을 설명해야 한다.

- 축약어보다 의미가 우선이다.
- 클래스명은 책임을 설명해야 한다.
- 패키지명은 계층과 역할을 설명해야 한다.
- `global`, `util`, `commonService` 같은 포괄적 이름은 지양한다.

예시:

- 좋음: `StockInfoLoader`, `StockPeriodCalculator`, `HolidayApiClient`
- 나쁨: `StockHelper`, `GlobalManager`, `logWarm`

## 리팩토링 판단 기준

아래 중 하나라도 해당하면 리팩토링 대상으로 본다.

- 계층 책임이 섞여 있다.
- 일관성이 없다.
- 같은 규칙이 여러 곳에 흩어져 있다.
- Service가 절차지향적으로 비대해졌다.
- DTO 변환이 Service에 퍼져 있다.
- 컨트롤러가 예외나 상태코드를 직접 조립한다.
- 외부 연동이 `support`나 도메인 계층에 섞여 있다.
- 배치 로직이 업무 로직과 뒤섞여 있다.

특히 일관성이 없는 부분은 반드시 리팩토링 대상에 포함한다.

## 테스트 규칙

리팩토링의 목표는 기능 변경이 아니라 구조 개선이다.
따라서 회귀 테스트는 필수다.

반드시 지켜야 하는 규칙:

- 리팩토링 전후 기능 동작은 같아야 한다.
- 예외 처리 구조를 바꾸더라도 비즈니스 의미는 같아야 한다.
- 기존 통합 테스트는 유지한다.
- 책임을 분리했다면 해당 객체의 단위 테스트를 추가한다.
- 최종 검증은 `./gradlew clean test` 통과다.

테스트 우선순위는 아래와 같다.

1. 기존 API 회귀 테스트
2. 새로 분리한 규칙 객체 단위 테스트
3. 예외 응답과 상태코드 테스트
4. 외부 연동 fallback 또는 파싱 테스트

## 기능 추가 체크리스트

새 기능을 추가할 때는 아래 순서로 점검한다.

1. 이 기능이 `core`, `storage`, `support`, `integration`, `batch` 중 어디에 속하는지 먼저 판단한다.
2. Controller, Service, 구현 협력 객체, 저장 계층의 책임이 섞이지 않았는지 확인한다.
3. 응답은 `ApiResponse`로 통일했는지 확인한다.
4. 실패는 `CoreException`과 `ErrorType`으로 표현했는지 확인한다.
5. DTO 변환을 Service에서 직접 하지 않았는지 확인한다.
6. 외부 연동이나 스케줄러가 잘못된 패키지에 들어가지 않았는지 확인한다.
7. 기존 동작이 바뀌지 않았는지 회귀 테스트로 검증한다.

## 리뷰 기준

코드 리뷰에서는 아래를 먼저 본다.

- 이 코드가 현재 패키지에 있는 이유가 명확한가
- Service가 얇게 유지되고 있는가
- 예외와 응답 규칙을 지켰는가
- 이름만 보고 책임이 추론되는가
- 일관성이 깨진 부분이 없는가
- 테스트가 구조 변경을 보호하고 있는가

## 최종 원칙

이 프로젝트의 백엔드는 빠르게 만드는 것보다 오래 유지되는 구조를 우선한다.

새 코드를 추가할 때는 기존 관성을 따라가는 것이 아니라,
이 문서의 규칙에 맞게 구조를 먼저 맞춘 뒤 구현한다.

구조가 어색하면 기능 추가 전에 먼저 구조를 바로잡는다.
일관성이 깨지면 작은 부분이라도 리팩토링 대상으로 본다.
