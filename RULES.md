# RULES.md - 프로젝트 표준 강제 규칙

본 문서는 코인 찌찍목 프로젝트의 아키텍처 안정성, 성능 최적화, 그리고 비즈니스 정합성을 보장하기 위해 **모든 AI 코딩 에이전트와 개발자가 반드시 준수해야 하는 강제(Hard) 규칙**입니다.

여기에 기재된 규칙은 코드를 수정하거나 설계를 변경할 때 최우선적으로 적용되며, 어떠한 예외도 허용하지 않습니다.

---

## 1. 데이터베이스 및 트랜잭션 관리 규칙

### Rule 1-1. 트랜잭션(tx) 최단 수명 유지 원칙
데이터베이스 트랜잭션 범위 안에서 느린 네트워크 I/O(외부 API 호출, Redis 접근 등)나 어플리케이션단 무거운 인메모리 연산(메모리 맵 삽입 등)을 수행하여 디비 커넥션을 오래 점유하는 행위를 원천 금지합니다.
* **지연 등록 적용**: 인메모리 캐시에 데이터를 삽입하거나 외부 비동기 이벤트를 발행하는 행위는 반드시 `TransactionSynchronizationManager.registerSynchronization`을 사용하여 **데이터베이스 커밋이 완전히 끝난 직후(`afterCommit`)에 실행**되도록 제어해야 합니다.

### Rule 1-2. Next-Key Lock 방지를 위한 인덱스 동치 매칭 원칙
InnoDB 격리 수준(REPEATABLE READ) 하에서 광범위한 `Next-Key Lock(갭 락)`이 유발되어 다른 회원의 거래/주문 트랜잭션이 차단되는 현상을 방지해야 합니다.
* **Record Lock 격하 강제**: 모든 비관적 락 조회(`SELECT ... FOR UPDATE`), 수정(UPDATE), 삭제(DELETE) 쿼리는 반드시 테이블의 **고유 제약조건(Unique Key / PK) 컬럼 전부를 등치(`=`) 조건**으로 매칭해야 합니다.
* 이를 통해 InnoDB의 잠금이 오직 대상 레코드 1행만 잠그는 **`Record Lock`**으로만 제한되도록 해야 합니다.
* **복합 인덱스 매칭**: 복합 Unique 제약 조건이 설정되어 있을 경우, 쿼리의 WHERE 조건에 복합 unique key 컬럼들이 순서대로 정확하게 명시되어 풀 테이블 스캔이 일어나지 않도록 검증해야 합니다.

## 2. 마켓 시세 및 웹소켓 구독 규칙

### Rule 2-1. 모의 투자 특화 시세 Ticker/Trade 역할 분리 원칙
우리는 외부 시세를 투명하게 추종하는 "모의 투자 플랫폼"이지만, 지정가/조건부 주문 체결 판단에는 실제 체결 가격 이동을 반영해야 합니다.
* **역할 분리 구독**: `ticker` 채널은 마크가, 지수가, 펀딩비, 신선도 판단의 기준으로 사용하고, `trade` 채널은 최근 체결가 이동 및 pending 주문 체결 트리거 판단에 사용합니다.
* **중복 소스 금지**: 같은 역할의 가격 데이터를 여러 채널에서 중복 판단하지 않습니다. 체결 트리거는 `trade` 이벤트 흐름을 기준으로 하고, 거래 가능 여부와 10초 신선도(Freshness) 타임아웃 검사는 안정 수신되는 `ticker` 패킷의 갱신 시간(`receivedAt`)을 기준으로 판단합니다.
* **오래된 시세 차단**: 10초 초과 시에는 결코 오래된 REST 복구 시세로 대체하지 않고 즉시 주문 트랜잭션을 안전하게 중단(롤백)시킵니다.

## 3. 코드 품질 및 클래스 설계 규칙

### Rule 3-1. 내부 클래스 외부 참조 금지 및 독립 분리 원칙
특정 클래스 내부에 선언된 내부 클래스(Inner Class, Static Nested Class 등)는 오직 그 외부 클래스 내부에서만 캡슐화되어 비공개(private) 혹은 제한적인 용도로만 사용되어야 합니다.
* **외부 참조 금지**: 다른 외부 클래스에서 특정 클래스의 내부 클래스를 직접 임포트(Import)하여 참조하거나, 외부 API의 반환 타입/파라미터 타입으로 노출하는 설계 구조를 철저히 금지합니다.
* **독립 클래스 분리**: 내부 클래스가 외부의 다수 컴포넌트 사이에서 공유되거나 재사용되어야 하는 요건이 생기면, 이는 캡슐화 위반 및 강한 결합도 신호이므로 **해당 클래스를 패키지 수준의 최상위 독립 클래스(Top-level Class)로 완전히 격상시켜 분리(Extract Class)해야 합니다.**

## 4. 규칙의 승격 및 추가 절차
* 개발 도중 발견된 우수 설계 기법이나 성능 병목 회피 전략은 즉시 `Precedent.md`에 선례(Precedent) 형태로 작성하여 수집합니다.
* 특정 선례가 **3회 이상 반복 적용**되었음이 검증되면, 해당 선례는 정식 규칙으로 승격되어 본 `RULES.md` 문서에 영구 추가됩니다.
