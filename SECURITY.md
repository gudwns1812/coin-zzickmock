# SECURITY.md

이 문서는 `coin-zzickmock` 저장소에서 보안과 관련된 구현, 리뷰, 리팩토링을 할 때 따르는 기준 문서입니다. 목적은 두 가지입니다.

첫째, 기능 구현 중 보안 사고로 이어질 수 있는 선택을 미리 막습니다. 둘째, 리뷰 시 "무엇을 확인해야 하는지"를 빠르게 판단할 수 있게 합니다. 이 문서는 사람이 읽기 쉬워야 하지만, 동시에 Codex 같은 코딩 에이전트가 바로 체크리스트처럼 사용할 수 있어야 합니다.

## 적용 범위

이 문서는 저장소 전체에 적용합니다.

- 프론트엔드: `frontend/`
- 백엔드: `backend/`
- 인증, 인가, 쿠키, 토큰, API 호출, 외부 연동, 로그, 설정값, 배포 설정

보안과 관련된 작업에서는 이 문서를 일반 구현 가이드보다 우선해서 해석합니다. 기능 요구사항과 충돌할 경우, 우선 안전한 방향으로 구현하고 필요한 예외는 코드와 PR 설명에 명시합니다.

## 코덱스를 위한 작업 원칙

보안 관련 작업에서 Codex는 아래 순서로 행동합니다.

1. 먼저 신뢰 경계를 확인합니다.
   사용자 브라우저, Next.js 서버, Spring Boot 서버, Redis/MySQL, 외부 API 중 어느 지점에서 입력이 들어오고 어디까지 신뢰할 수 있는지 먼저 적어야 합니다.
2. 인증과 인가를 분리해서 판단합니다.
   "로그인했는가"와 "이 작업을 할 권한이 있는가"는 별개로 확인합니다.
3. 입력 검증과 출력 노출을 동시에 봅니다.
   입력이 안전한지뿐 아니라, 응답과 로그에 민감한 값이 새지 않는지도 확인합니다.
4. 프론트엔드 보호만으로 끝내지 않습니다.
   `middleware.ts` 같은 프론트 보호는 사용자 경험용 보조 장치일 뿐이며, 최종 권한 검사는 반드시 서버에서 다시 해야 합니다.
5. 보안상 애매하면 더 엄격한 쪽을 선택합니다.
   "아마 괜찮다"는 판단으로 merge하지 않습니다.

## 핵심 보안 원칙

### 1. 클라이언트는 신뢰하지 않습니다

브라우저 값, 쿼리스트링, pathname, local state, hidden input, 쿠키 값은 모두 변조 가능하다고 가정합니다. 프론트엔드에서 막았더라도 서버는 동일한 검증을 다시 해야 합니다.

예를 들어 `frontend/middleware.ts`에서 특정 경로 접근을 막아도, 실제 데이터 조회나 주문 실행 같은 민감한 동작은 백엔드에서 다시 인증과 인가를 검사해야 합니다.

### 2. 인증과 인가는 분리합니다

인증은 사용자가 누구인지 확인하는 과정이고, 인가는 그 사용자가 지금 이 행동을 해도 되는지 판단하는 과정입니다. 로그인 여부만 확인하고 리소스 소유권, 역할, 범위를 확인하지 않는 구현은 불완전합니다.

예를 들어 "내 포트폴리오 조회" 기능은 로그인만으로 충분하지 않을 수 있습니다. 요청한 리소스가 정말 현재 사용자 소유인지도 확인해야 합니다.

### 3. 토큰을 "읽는 것"과 "검증하는 것"은 다릅니다

JWT를 decode하는 것은 단지 내용을 읽는 행위일 뿐이며, 서명 검증이 아닙니다. 따라서 decode 결과만 믿고 인증 완료로 간주하면 안 됩니다.

현재 `frontend/utils/auth.ts`는 `jwtDecode`를 사용하고 있으므로, 이 코드는 사용자 경험을 위한 보조 판단에는 쓸 수 있어도 최종 인증 근거로 사용하면 안 됩니다. 서버 권한 검사는 반드시 서버가 서명, 만료 시간, 발급자, 대상 값을 검증한 뒤 수행해야 합니다.

### 4. 민감 정보는 기본적으로 숨깁니다

다음 값은 로그, 응답, 클라이언트 상태, 에러 메시지에 그대로 노출하지 않습니다.

- 액세스 토큰, 리프레시 토큰, 세션 ID
- 비밀번호, 인증 코드, 개인 식별 정보
- 내부 호스트, DB 접속 정보, 서드파티 비밀키
- 결제, 계좌, 거래 관련 민감 데이터

개발 편의를 이유로 민감 값을 콘솔에 찍는 코드는 금지합니다.

### 5. 안전한 기본값을 사용합니다

설정을 정하지 않았을 때 더 안전한 쪽으로 동작해야 합니다. 예를 들면 다음과 같습니다.

- 권한 정보가 불분명하면 허용이 아니라 거부
- 토큰 검증 실패 시 계속 진행이 아니라 로그아웃 또는 401/403
- 선택적 검증보다는 필수 검증
- 넓은 공개 범위보다 최소 공개 범위

## 구현 규칙

### 인증과 세션

- 인증 상태는 서버가 최종 판단합니다.
- 토큰이 있다면 서명, 만료 시간, 발급자, 대상, 토큰 종류를 검증해야 합니다.
- 액세스 토큰과 리프레시 토큰의 용도는 분리합니다.
- 쿠키 기반 인증이라면 `HttpOnly`, `Secure`, `SameSite` 정책을 명시합니다.
- 로그아웃은 클라이언트 상태 삭제만으로 끝내지 말고 서버 기준 세션 무효화 또는 쿠키 만료 처리까지 포함합니다.
- 토큰 검증 실패를 "게스트처럼 계속 사용"으로 넘길 수 있는지 신중히 판단합니다. 민감 기능에서는 반드시 차단합니다.

금지 패턴:

- decode만 하고 검증 없이 인증 완료로 간주
- 만료 시간을 확인하지 않음
- 토큰 종류를 구분하지 않음
- 브라우저 저장소에 장기 토큰을 평문 저장

### 인가

- 사용자 본인 데이터인지 항상 확인합니다.
- 관리자 기능, 운영 기능, 배치 기능은 일반 사용자 흐름과 분리합니다.
- 프론트엔드에서 버튼을 숨기는 것은 인가가 아닙니다.
- "ID만 알면 다른 사용자의 데이터 조회 가능"한 구조가 생기지 않게 합니다.

리뷰할 때 반드시 보는 항목:

- 요청 파라미터에 사용자 ID가 들어갈 때, 서버가 현재 로그인 사용자와 대조하는가
- 목록 API가 다른 사용자의 데이터까지 노출할 가능성은 없는가
- 운영자 전용 API가 공개 라우트에 섞여 있지 않은가

### 입력 검증

- 경로 파라미터, 쿼리스트링, 헤더, 쿠키, request body는 모두 검증합니다.
- 형식 검증과 의미 검증을 분리합니다.
  형식 검증은 "숫자인가, 길이가 맞는가"이고, 의미 검증은 "존재 가능한 값인가, 현재 사용자에게 허용된 값인가"입니다.
- 허용 목록 방식이 가능하면 허용 목록을 우선합니다.
- 정렬 키, 필터 키, 필드명처럼 동적으로 들어오는 값은 서버에서 화이트리스트로 제한합니다.

예를 들어 `frontend/middleware.ts`에서 종목 코드를 6자리 숫자로 제한하는 것은 좋은 1차 방어입니다. 하지만 서버에서도 동일한 형식과 존재 여부를 다시 검증해야 합니다.

### 출력과 오류 처리

- 사용자에게는 필요한 최소한의 오류만 반환합니다.
- 내부 스택 트레이스, SQL 에러, 토큰 파싱 실패 원문, 외부 API 비밀값은 응답에 담지 않습니다.
- 에러 응답 포맷은 일관되게 유지합니다.
- 보안상 의미 있는 실패는 서버 로그에 남기되, 민감 값은 마스킹합니다.

좋은 예:

- "인증이 필요합니다"
- "요청 형식이 올바르지 않습니다"
- "해당 리소스에 접근할 수 없습니다"

나쁜 예:

- "JWT signature does not match locally computed signature ..."
- "DB password for writer cluster is ..."

### 비밀값과 설정

- 비밀값은 코드에 하드코딩하지 않습니다.
- `NEXT_PUBLIC_*` 환경 변수에는 비밀값을 넣지 않습니다.
- 환경 변수 이름만으로 용도를 구분할 수 있게 작성합니다.
- 샘플 설정 파일에는 실제 키가 아니라 플레이스홀더를 넣습니다.
- 테스트 코드에도 운영 비밀값을 복사하지 않습니다.

체크 포인트:

- `process.env` 값이 클라이언트 번들로 노출되지 않는가
- 예제 코드에 실제 토큰, 쿠키, 키 문자열이 박혀 있지 않은가
- 에러 메시지에 설정값이 직렬화되어 찍히지 않는가

### 프론트엔드 보안

- 브라우저에서 받은 인증 상태는 참고 정보로만 사용합니다.
- 민감 데이터는 가능한 한 서버 컴포넌트 또는 서버 API에서 처리합니다.
- 사용자 입력을 그대로 HTML로 삽입하지 않습니다.
- 외부 URL 리다이렉트가 있다면 허용 도메인 검증 또는 상대 경로 제한을 둡니다.
- Sentry, analytics, console 로그에 민감 정보가 실리지 않게 합니다.

프론트엔드에서 자주 놓치는 항목:

- `console.log(token)` 같은 디버깅 코드
- URL 파라미터에 액세스 토큰 전달
- 에러 화면에서 서버 응답 전체를 그대로 렌더링
- 권한 체크를 오직 `middleware.ts`에만 의존

### 백엔드 보안

- 모든 민감 API는 서버 인증과 인가를 통과해야 합니다.
- DTO 검증과 도메인 규칙 검증을 분리합니다.
- 저장 전 데이터 정규화와 검증을 명시합니다.
- 외부 연동 응답도 신뢰하지 말고 필요한 필드만 추려 사용합니다.
- 배치 작업과 내부 작업도 "내부라서 안전하다"고 가정하지 않습니다.

백엔드에서 자주 놓치는 항목:

- 테스트용 우회 인증 코드가 운영 코드에 남음
- 관리자/내부 API를 공용 포트에 그대로 노출
- 예외 메시지에 엔티티 전체를 포함
- 로그에 request body 전체 출력

### 데이터 저장과 로그

- 꼭 필요한 데이터만 저장합니다.
- 나중에 쓰지 않을 민감 정보는 저장하지 않습니다.
- 로그는 디버깅 가능성과 정보 노출 위험 사이에서 최소 수집 원칙을 지킵니다.
- 토큰, 쿠키, Authorization 헤더, 주민성 정보, 결제 정보는 로그에서 마스킹하거나 제외합니다.

권장 방식:

- 사용자 식별은 내부 ID 사용
- 원문 대신 해시 또는 마스킹값 사용
- 운영 로그와 개발 로그의 상세 수준 분리

### 외부 연동

- 외부 API 입력과 응답은 모두 비신뢰 입력으로 취급합니다.
- 타임아웃, 재시도, 실패 처리를 명시합니다.
- 외부 서비스 장애 시 민감 정보가 에러 메시지로 흘러나오지 않게 합니다.
- 웹훅, 콜백이 있다면 서명 검증 또는 출처 검증을 포함합니다.

## 구현 전에 확인할 질문

보안이 걸린 작업을 시작하기 전에 아래 질문에 답해야 합니다.

- 이 기능의 신뢰 경계는 어디인가
- 사용자가 조작할 수 있는 입력은 무엇인가
- 서버가 최종 인증을 어디서 검증하는가
- 서버가 최종 인가를 어디서 검증하는가
- 민감 정보가 응답, 로그, 클라이언트 번들에 노출될 가능성은 없는가
- 실패 시 기본 동작은 허용인가 거부인가
- 테스트로 무엇을 증명할 것인가

이 질문에 답할 수 없다면 구현을 미루고 먼저 설계를 명확히 해야 합니다.

## 리뷰 체크리스트

리뷰어와 Codex는 아래 항목을 기본 체크리스트로 사용합니다.

### 인증

- 토큰 decode와 검증을 혼동하지 않았는가
- 만료, 서명, 발급자, 대상, 토큰 종류 검증이 있는가
- 쿠키 속성이 안전하게 설정되어 있는가

### 인가

- 로그인 여부만 확인하고 권한 검사를 생략하지 않았는가
- 다른 사용자의 리소스를 ID만으로 조회/수정할 수 없게 되어 있는가
- 관리자 기능이 일반 사용자 흐름과 분리되어 있는가

### 입력 검증

- path, query, body, header, cookie를 모두 검증하는가
- 동적 필드명이나 정렬 키를 화이트리스트로 제한하는가
- 프론트 검증과 별개로 서버 검증이 있는가

### 데이터 노출

- 응답에 내부 구현 정보나 민감 정보가 새지 않는가
- 로그에 토큰, 쿠키, 비밀번호, 개인 정보가 남지 않는가
- 분석 도구와 모니터링 도구에 민감 정보가 전송되지 않는가

### 설정과 비밀값

- 비밀값이 코드, 테스트, 샘플 파일에 하드코딩되지 않았는가
- `NEXT_PUBLIC_*`에 비밀값이 들어가지 않았는가
- 운영/개발 설정이 섞이지 않았는가

### 회귀 위험

- 보안을 위해 추가한 로직이 실패 시 우회 가능하지 않은가
- 테스트가 성공 경로뿐 아니라 실패 경로도 검증하는가
- 임시 우회 코드, 디버그 코드, 완화용 플래그가 남아 있지 않은가

## 테스트 기준

보안 관련 변경에는 가능하면 아래 테스트 중 해당되는 것을 포함합니다.

- 인증 성공 테스트
- 인증 실패 테스트
- 인가 실패 테스트
- 잘못된 입력 형식 테스트
- 다른 사용자 리소스 접근 차단 테스트
- 로그/응답에 민감 정보가 없는지 확인하는 테스트 또는 수동 검증

최소 기준은 "성공 경로 1개 + 실패 경로 1개"입니다. 민감 기능이면 더 강하게 요구합니다.

## 즉시 중단하고 다시 판단해야 하는 경우

아래 상황이면 바로 구현을 멈추고 더 안전한 설계를 선택합니다.

- JWT를 decode만 하고 인증 완료로 처리하려는 경우
- 프론트엔드 조건 분기만으로 권한 제어를 끝내려는 경우
- 민감 값을 로그나 에러에 그대로 찍으려는 경우
- 비밀값을 코드나 공개 환경 변수에 넣으려는 경우
- 다른 사용자의 데이터 접근 가능성을 완전히 배제하지 못한 경우
- "일단 붙이고 나중에 보안 처리" 방식으로 merge하려는 경우

## 이 저장소에 대한 추가 메모

현재 저장소에는 Next.js 프론트엔드와 Spring Boot 백엔드가 함께 존재합니다. 따라서 보안 검토 시 다음 해석을 기본값으로 사용합니다.

- `frontend/middleware.ts`의 라우트 차단은 보조 장치이며, 최종 보안 장치는 아닙니다.
- `frontend/utils/auth.ts`처럼 브라우저 또는 프론트 서버에서 토큰 내용을 읽는 코드는 인증 보조 정보일 뿐이며, 최종 서버 인증의 근거가 아닙니다.
- `backend/`에 인증/인가 로직이 추가되거나 복구될 때는 이 문서 기준으로 서버 검증을 우선 설계합니다.

## 문서 유지 원칙

- 보안 사고로 이어질 수 있는 새로운 패턴을 발견하면 이 문서에 바로 추가합니다.
- 반복해서 지적되는 리뷰 항목은 체크리스트에 승격합니다.
- 저장소 구조가 크게 바뀌면 예시 경로를 최신 상태로 갱신합니다.
- 이 문서는 한국어로 유지하되, 코드 식별자와 설정 키는 원문 표기를 사용합니다.
