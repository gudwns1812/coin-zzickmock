# 코인 선물 모의투자 플랫폼 설계 초안

## 목적

이 문서는 `coin-zzickmock`를 "주식 조회 UI 일부를 재사용한 상태"에서
"코인 선물 모의투자 플랫폼"으로 전환하기 위한 첫 제품 설계 초안이다.

이 문서의 목표는 세 가지다.

1. 무엇을 만들지 범위를 먼저 고정한다.
2. 코인 선물 도메인에서 반드시 필요한 개념을 주식 도메인과 분리해서 정의한다.
3. 다음 단계에서 화면 설계, 백엔드 설계, ExecPlan으로 내려갈 수 있는 공통 기준을 만든다.

## 제품 한 줄 정의

Bitget 선물 시장 데이터를 바탕으로 사용자가 가상 자산으로 포지션을 열고, 레버리지와 마진, 청산, 펀딩비를 경험할 수 있는 데스크톱 우선 모의투자 플랫폼.

중요:

- 이 제품은 우선 `실거래`가 아니라 `모의투자`를 목표로 한다.
- 실제 자산 수탁, 실주문 전송, 현금 입출금은 MVP 범위에 포함하지 않는다.
- 실제 시장 데이터는 사용할 수 있지만, 주문 체결과 손익 반영은 우리 서비스의 시뮬레이션 규칙에 따라 처리한다.

## 왜 이 제품을 만드는가

현재 저장소는 주식 조회/포트폴리오 UI 조각을 일부 보유하고 있지만,
코인 선물의 핵심 경험인 아래 요소는 아직 없다.

- 롱/숏 포지션
- 레버리지 설정
- 격리/교차 마진
- 강제 청산 가격
- 펀딩비 반영
- 미체결 주문과 조건부 주문

즉, 기존 UI를 그대로 치환하는 수준이 아니라
"시장 조회 서비스"에서 "선물 트레이딩 시뮬레이터"로 제품의 중심을 옮겨야 한다.

## 목표 사용자

### 1. 입문형 사용자

- 코인 선물 개념은 알고 있지만 실제 돈으로 바로 거래하기는 부담스럽다.
- 포지션, 청산, 펀딩비를 안전하게 연습하고 싶다.

### 2. 복습형 사용자

- 이미 현물/선물 경험이 있다.
- 특정 마켓(BTCUSDT, ETHUSDT 등)의 움직임에 맞춰 전략을 시험해 보고 싶다.

### 3. 콘텐츠형 사용자

- 자신의 거래 기록, 수익률, 승률, 리스크를 시각적으로 보고 싶다.
- 향후 리더보드, 챌린지, 전략 회고로 확장될 수 있다.

## 제품 원칙

### 1. 선물의 핵심 개념을 숨기지 않는다

단순 가격 조회 앱이 아니라, 포지션과 리스크를 이해시키는 경험이어야 한다.
그래서 주문 입력보다 아래 정보를 먼저 잘 보여줘야 한다.

- 현재 가격
- Mark Price
- Funding Rate
- 포지션 크기
- 사용 증거금
- 예상 청산가
- 미실현 손익

### 2. 모의투자라도 규칙은 느슨하지 않게 간다

사용자가 학습용으로 신뢰할 수 있으려면 손익, 수수료, 펀딩비, 청산 규칙이 일관돼야 한다.
"대충 비슷한 느낌"보다 "설명 가능한 시뮬레이션 규칙"이 중요하다.

### 3. 데스크톱 우선 대시보드 경험을 유지한다

현재 프론트는 데스크톱 우선 금융 UI 구조를 이미 갖고 있다.
MVP에서는 이 강점을 살려 모바일 최적화보다 데스크톱 사용성을 우선한다.

### 4. 주식 자산은 레이아웃만 재사용하고 도메인은 새로 세운다

기존 `stock`, `portfolio`, `interest` 구조는 이름부터 선물 도메인과 어긋난다.
레이아웃, 카드, 차트 컨테이너, 사이드바 패턴은 일부 재사용 가능하지만
API 타입, 훅, 상태 이름은 코인 선물 기준으로 다시 정의한다.

## 현재 결정된 운영 규칙

아래 항목은 현재 사용자와 합의된 1차 고정값이다.

- 시세/선물 기준 거래소는 `Bitget`으로 한다.
- 지원 심볼은 우선 `BTCUSDT`, `ETHUSDT` 두 개만 연다.
- 회원가입과 로그인을 MVP에 포함한다.
- 초기 지급 잔고는 계정당 `100000 USDT`다.
- 레버리지는 최대 `50x`까지 허용한다.
- 마진 모드는 `ISOLATED`, `CROSS`를 모두 지원한다.
- 수수료는 `maker 0.015%`, `taker 0.05%`로 고정한다.
- 상점(`shop`)과 포인트 적립을 MVP 범위에 넣는다.
- 리더보드/대회는 후순위로 미룬다.

## MVP 범위

### 포함

1. 회원가입과 로그인
2. 가상 USDT 시드머니 지급
3. Bitget 기반 주요 선물 마켓 목록 조회
4. 심볼 상세 조회
5. 캔들 차트와 기본 시장 지표 표시
6. 시장가/지정가 주문
7. 롱/숏 포지션 오픈과 종료
8. 레버리지 설정
9. 격리 마진과 교차 마진 지원
10. 포지션별 미실현 손익, 실현 손익, 청산가 계산
11. 주문/체결/거래 히스토리
12. 개인 대시보드
13. 관심 심볼 목록
14. 포인트 적립과 상점
15. 메인 대시보드의 읽기 전용 실현 수익률 랭킹

### 제외

1. 실제 거래소 주문 연동
2. 원화/달러 실입금 및 출금
3. 조건부 주문, TP/SL, 트레일링 스탑
4. 다중 자산 담보
5. 소셜 기능, 경쟁전, 대회, 기간별/주변 순위 같은 확장 리더보드
6. 모바일 앱 수준의 반응형 완성도

## 핵심 사용자 흐름

### 1. 온보딩

사용자는 회원가입 또는 로그인으로 진입한다.
회원가입 직후 가상 `100000 USDT` 잔고와 기본 관심 심볼 세트를 지급한다.

메인 대시보드의 랭킹 수익률도 이 초기 지급액을 분모로 사용한다.
회원가입으로 기본 `TradingAccount`가 생성되면 랭킹 Redis ZSET에도 신규 멤버를 추가한다.

### 2. 마켓 탐색

사용자는 마켓 목록에서 `BTCUSDT`, `ETHUSDT` 같은 심볼을 탐색한다.
각 항목에는 현재가, 24h 변동률, 거래량, 펀딩비, 오픈 인터레스트 요약을 보여준다.
대시보드 랭킹은 열린 포지션의 미실현 손익을 포함하지 않고, 포지션 종료와 수수료가 반영된 `trading_accounts.wallet_balance` 기준으로 산출한다.
랭킹 ZSET은 가입/지갑 잔고 변경 이벤트가 발생할 때 해당 멤버만 즉시 갱신하고, 스케줄러는 1시간 단위 전체 재집계로 누락을 보정한다.

### 3. 심볼 상세

사용자는 특정 심볼 상세 화면에서 다음을 본다.

- 가격/등락/거래량
- 캔들 차트
- 호가 또는 요약 체결 흐름
- 펀딩비, Mark Price, Index Price
- 내 보유 포지션과 미체결 주문
- 주문 패널

### 4. 주문 실행

사용자는 주문 패널에서 아래를 입력한다.

- 롱 또는 숏
- 시장가 또는 지정가
- 수량
- 레버리지
- 격리 또는 교차 마진

주문 전 예상 사용 증거금, 청산가, 수수료를 보여준다.

### 5. 포지션 관리

사용자는 대시보드에서 열린 포지션, 미실현 손익, 위험도를 확인한다.
포지션 일부 또는 전체를 종료할 수 있다.

### 6. 회고

사용자는 주문/체결/손익 히스토리를 보며 자신의 성과를 복기한다.
MVP에서는 기본 리스트와 요약 카드 중심으로 시작한다.

## 핵심 화면 구조

### 1. `/markets`

기존 `/stock`를 대체하는 메인 마켓 대시보드.

- 상단: 시장 개요 카드
- 좌측/중앙: 주요 심볼 리스트, 카테고리 또는 테마
- 우측: 관심 심볼, 내 계정 요약, 위험 요약

### 2. `/markets/[symbol]`

기존 `/stock/[code]`를 대체하는 심볼 상세 화면.

- 상단: 심볼 헤더와 주요 지표
- 중앙: 차트
- 우측: 주문 패널
- 하단: 포지션, 미체결 주문, 최근 체결

### 3. `/mypage`

계정 영역은 `/mypage` route family로 재구성한다.
`/portfolio`는 호환성 리다이렉트로 남긴다.

- `/mypage`:
  사용자 기본 정보, 지갑 잔고, 사용 가능 잔고, 열린 포지션 수, 포인트
- `/mypage/assets`:
  assets 스타일 총자산 패널, 사용 가능 잔고, KST 일별 net realized PnL 캘린더
- `/mypage/points`:
  현재 포인트와 point history

### 4. `/watchlist`

기존 관심 종목 그룹 기능을 관심 심볼 목록으로 단순화.

### 5. `/shop`

포인트 사용처인 상점 화면.

- 포인트 잔액
- DB에서 관리되는 상품 목록
- 커피 교환권 신청 모달
- sold-out, 구매 제한, 포인트 부족 상태
- 신청 성공/실패 피드백

### 6. `/learn` 또는 온보딩 모달

선물 용어 설명과 리스크 안내.
MVP에서는 전용 페이지보다 진입 모달이나 도움말 패널로 시작 가능하다.

## 시뮬레이션 규칙 초안

### 가격 기준

- 차트와 시세 표시는 외부 시장 데이터 기반
- 포지션 평가와 청산 계산은 `mark price` 기준
- 시장가 주문 체결은 최신 체결가 기반 taker 체결로 처리한다
- 지정가 주문은 최신 체결가가 걸린 가격을 넘어서거나 같아지는 시점에 체결 대상으로 본다
- 제출 즉시 체결 가능한 지정가 주문은 taker로 처리하고, 대기 후 체결되면 maker로 처리한다

### 수수료

- `maker 0.015%`
- `taker 0.05%`
- 주문 전 예상 수수료를 항상 노출

### 펀딩비

- 우선 8시간 단위 정산 모델을 가정
- 오픈 포지션에만 적용
- 히스토리에 별도 ledger로 남긴다

### 청산

- 격리와 교차 마진을 모두 지원한다
- 포지션별 유지 증거금률을 기준으로 청산가 계산
- 청산 발생 시 잔여 증거금 반영 규칙을 명시적으로 남긴다

### 포인트

- 유저가 실현 수익을 확정하면 수익 금액 구간에 따라 포인트를 지급한다
- 포인트는 현금성 출금이 아니라 상점 소비용 내부 재화다
- 포인트 지급 규칙은 악용 방지를 위해 실현 손익 기준으로만 계산한다
- 포인트는 정수 단위로 저장하고 차감/환불도 정수 단위로 처리한다
- 교환권 신청은 포인트를 즉시 차감하고 point history에 `REDEMPTION_DEDUCT`를 남긴다
- 발송 전 취소/환불은 포인트를 정확히 한 번 복구하고 `REDEMPTION_REFUND`를 남긴다

정확한 수식은 후속 설계 문서에서 고정하되,
지금 단계에서 중요한 것은 "모든 손익/위험 계산이 포지션 엔진에서 일관되게 나온다"는 구조다.

## 시스템 구조 제안

### 프론트엔드

- Next.js App Router 유지
- 메인 라우트는 `markets`, `mypage`, `watchlist`, `shop` 기준으로 재편
- Server Component 우선 원칙 유지
- 시장 데이터와 사용자 포지션 데이터는 React Query로 갱신
- 글로벌 상태는 사이드바, 필터, 임시 입력 같은 UI 상태 위주로만 유지

### 백엔드

Spring Boot를 단일 백엔드로 유지하되 아래 feature를 우선 만든다.

1. `feature/market`
   마켓 목록, 심볼 상세, 캔들, 펀딩비, mark price
2. `feature/account`
   사용자 계정, 잔고, 시드머니, 계정 요약
3. `feature/order`
   주문 생성, 취소, 조회
4. `feature/position`
   포지션 생성, 평가, 종료, 청산
5. `feature/history`
   체결/손익/펀딩비/수수료 히스토리
6. `feature/reward`
   포인트 적립, point ledger, DB-managed shop item, coffee voucher redemption, admin processing, notification boundary

### 외부 데이터 연결

- 거래 실행은 모의지만, 시세 데이터는 Bitget에서 받는다
- 외부 API 연결은 `Providers -> ConnectorProvider` 뒤로 숨긴다
- 초기 MVP는 polling + Redis 캐시 구조로도 충분하다
- 후속 단계에서 WebSocket ingestion을 도입할 수 있다

### 저장소

- MySQL: 계정, 주문, 포지션, 히스토리, 포인트, 상점 구매 이력
- Redis: 실시간 시세 캐시, 짧은 TTL 스냅샷, 이벤트성 데이터

## 현재 프론트 자산의 재사용 판단

### 재사용 가능

- App Router 기반 레이아웃 구조
- 사이드바/헤더/카드형 대시보드 배치
- 차트 컨테이너와 공용 UI 조합
- React Query, Zustand, MSW 기반 개발 경험

### 이름만 바꿔서는 안 되는 것

- `stocks.ts`, `portfolio.ts` API 함수

- `type/stocks/*`
- 주식 시장 영업시간 가정 로직

기존 파일을 무리하게 덮어쓰기보다,
`market`, `futures`, `position` 같은 새 도메인 이름으로 병행 생성한 뒤
기존 주식 전용 코드는 점진적으로 제거한다.

## 보안/신뢰 경계

### 신뢰 경계

- 브라우저의 주문 입력값은 신뢰하지 않는다
- 레버리지, 수량, 주문 가격은 서버에서 다시 검증한다
- 계정 소유권과 포지션 접근 권한은 서버가 최종 판단한다

### 민감 포인트

- JWT는 decode만으로 신뢰하면 안 된다
- 주문 실행 API는 인증과 인가를 모두 분리해서 검증해야 한다
- 손익/청산 계산은 프론트에서 확정하지 않고 서버 결과를 기준으로 한다

### 운영 안전장치

- 모의투자임을 UI 전반에 명확히 표시
- 투자 권유 표현보다 학습/시뮬레이션 성격을 강조
- 외부 시세 장애 시 주문 기능을 제한하거나 읽기 전용 모드로 전환

## 단계별 구현 로드맵

### Phase 1. 시장 조회 전환

- 기존 `/stock` 흐름을 `/markets` 개념으로 전환
- Bitget `BTCUSDT`, `ETHUSDT` 시세 연결
- 심볼, 현재가, 차트, 펀딩비 표시
- 주식 타입과 명칭 제거

### Phase 2. 계정과 포지션 MVP

- 회원가입/로그인
- 가상 `100000 USDT` 지갑
- 포지션 오픈/종료
- 격리/교차 마진
- 최대 `50x` 레버리지
- 손익 계산
- 포트폴리오를 선물 계정 대시보드로 전환

### Phase 3. 주문/히스토리 완성

- maker/taker 수수료 적용
- 지정가 주문 체결 규칙 적용
- 미체결 주문 조회/취소
- 체결/손익/펀딩비 히스토리

### Phase 4. 보상과 상점

- 실현 손익 기반 포인트 적립
- point ledger
- shop 아이템과 구매 처리

### Phase 5. 리스크 고도화

- 청산 엔진 정교화
- 유지 증거금 구간
- 경고 배지와 위험도 시각화

   `markets`, `symbol detail`, `portfolio`별 컴포넌트/데이터 요구사항
2. 선물 계산 규칙 문서
   손익, 수수료, 펀딩비, 청산 수식과 예시
3. 구현용 ExecPlan
   어떤 순서로 파일과 기능을 만들지 단계별로 기록

## 현재 남은 열린 질문

아래는 아직 후속 설계에서 더 정해야 하는 항목이다.

1. 회원가입 방식이 이메일 기반인지, OAuth를 같이 붙일지
2. 교차 마진의 청산 계산을 계정 단위로 어디까지 현실적으로 반영할지
3. 지정가 주문 체결 시 슬리피지와 부분 체결을 MVP에서 얼마나 단순화할지
4. 포인트 지급 구간과 상점 아이템 종류를 어떻게 설계할지
5. Bitget 시세 수집을 polling으로만 시작할지, 초기에 WebSocket까지 붙일지
