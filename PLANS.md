# 코덱스 실행 계획서(ExecPlan) 작성 규칙

이 문서는 이 저장소에서 사용하는 실행 계획서(이하 "ExecPlan")의 작성 규칙이다. ExecPlan은 "계획"이면서 동시에 "실행 가능한 명세서"다.

독자는 이 저장소를 전혀 모르는 초보자라고 가정한다. 독자가 가진 것은 현재 작업 트리와 ExecPlan 파일 하나뿐이며, 이전 계획에 대한 기억도 없고 외부 문맥도 없다. 따라서 ExecPlan은 반드시 자기완결적이어야 한다.

이 저장소의 ExecPlan 규칙은 plan 스킬이 요구하는 결과물 형태(요구 사항 요약, 테스트 가능한 수용 기준, 구현 단계, 위험/완화, 검증 절차)를 ExecPlan의 필수 구성 요소로 포함한다. 강한 제약(자기완결성, 검증 가능성, 살아 있는 문서, 한국어 작성)은 그대로 유지한다.

## 문서의 위치와 생명주기

- 승인 전 초안: 임시 위치에 있어도 된다.
- 승인된 ExecPlan: 반드시 `docs/exec-plans/active/` 아래에 둔다.
- 현재 범위를 더 이상 진행할 일이 없을 때: 동일 파일을 `docs/exec-plans/completed/`로 이동한다.

여기서 "현재 범위"는 Codex 세션 전체가 아니라 ExecPlan 하나를 뜻한다. 완료 판단 기준은 "이 계획으로 더 이상 진행할 일이 없는지"이며, `진행 현황`에 남은 항목이 없으면 completed로 이동한다. 구현이 어느 정도 끝났다는 이유만으로 옮기지 않는다.

## 작성 워크플로

1. 새 계획을 만들거나 기존 계획의 큰 방향을 다시 잡을 때는 Plan mode로 초안을 만든다.
2. 초안은 반드시 `docs/exec-plans/plan-template.md`에서 시작해 현재 작업에 맞게 채운다.
3. 사용자가 승인하면 계획 파일을 `docs/exec-plans/active/`로 옮기고, 구현/검증/기록을 진행하면서 계속 갱신한다.

구현 단계에서는 사용자에게 "다음 단계"를 묻지 말고 ExecPlan의 다음 항목으로 곧바로 진행한다. 멈추는 시점마다 `진행 현황`을 갱신하고, 필요한 경우 항목을 더 쪼개 "완료된 부분"과 "남은 부분"이 드러나게 만든다.

## 언어 규칙

- ExecPlan은 반드시 한국어로 작성한다.
- 코드/명령어/경로/타입 이름처럼 원문 유지가 필요한 식별자만 원문 표기를 유지한다.
- 필수 섹션 이름은 아래 "필수 섹션"에 나온 한국어 제목을 그대로 사용한다.

## 협상 불가 요구 사항

- 자기완결적이어야 한다: 이 문서 하나만으로 초보자가 작업을 시작하고 끝까지 완료할 수 있어야 한다.
- 살아 있는 문서여야 한다: 진행/발견/결정이 생길 때마다 즉시 업데이트한다.
- 관찰 가능한 결과 중심이어야 한다: 사용자가 무엇을 할 수 있게 되는지, 무엇을 실행하고 무엇을 관찰해야 하는지를 구체적으로 적는다.
- 검증 가능해야 한다: 수용 기준은 테스트 가능한 형태로 쓴다(입력/출력/관찰 기준이 있어야 한다).
- 모호함을 남기지 않는다: 중요한 결정을 독자에게 떠넘기지 않는다. 애매하면 문서 안에서 방향을 정하고 근거를 남긴다.

## 형식 규칙

- ExecPlan은 일반 Markdown으로 작성한다.
- 다른 문서에 ExecPlan을 "삽입"해야 할 특별한 이유가 있을 때만, `md` 라벨이 붙은 단일 fenced code block 하나로 감싸서 제공한다.
- 체크박스(체크리스트)는 `진행 현황` 섹션에서만 사용한다. 그 외 섹션은 문장 중심으로 쓴다.
- 수용 기준과 검증 절차는 반드시 재현 가능한 입력/출력(또는 관찰 기준)을 포함해야 한다.
- 작업 계획과 구체적인 단계에는 가능한 한 구체적인 파일 경로를 포함한다.

## 필수 섹션(반드시 포함)

아래 섹션은 순서가 바뀌어도 되지만, 모두 존재해야 한다. 템플릿은 `docs/exec-plans/plan-template.md`를 따른다.

### 1) 목적 / 큰 그림

- 왜 필요한지
- 완료 후 사용자가 무엇을 할 수 있는지
- 무엇으로 성공을 확인할지(관찰 가능해야 함)

### 2) 범위

- 이번에 하는 것(in scope)
- 이번에 하지 않는 것(out of scope)
- 범위 밖이지만 관련 있는 후속 작업(있다면)

### 3) 진행 현황

체크박스 목록으로 세부 단계를 추적한다. 작업을 멈추는 모든 시점은 여기에 반영되어야 한다. 완료 항목에는 타임스탬프를 붙인다.

예:

- [x] (2026-04-20) 계획 초안 작성
- [ ] 구현
- [ ] 검증

### 4) 요구 사항 요약

요구 사항을 짧은 문장으로 요약한다. 기능 요구 사항과 비기능 요구 사항(성능/안전/장애 대응 등)을 구분해도 된다.

### 5) 수용 기준(테스트 가능한 형태)

각 항목은 반드시 검증 가능한 형태여야 한다.

- 어떤 명령 또는 입력을 준다
- 무엇이 출력/로그/상태로 관찰된다
- 실패하면 무엇을 확인해야 한다

### 6) 맥락과 길잡이

독자가 아무것도 모른다고 가정하고 다음을 적는다.

- 관련 문서(저장소 내 경로)
- 관련 코드 경로(최소 3개, 전체 경로로)
- 선행 조건(환경 변수, 로컬 실행 방법, 데이터 준비 등)
- 낯선 용어 정의(등장하면 즉시 정의)

외부 링크를 "참고하라"로 넘기지 않는다. 필요한 핵심 지식은 이 문서 안에 직접 풀어 쓴다.

### 7) 작업 계획

무엇을 어떤 순서로 바꿀지 문장으로 설명한다. 각 변경은 파일 경로 단위로 연결되어야 한다.

### 8) 구체적인 단계

초보자가 그대로 따라 할 수 있게 "어디에서 무엇을 실행하는지"를 단계별로 적는다. 명령이 출력을 만든다면 짧은 예상 예시를 함께 적어도 된다.

### 9) 위험과 완화

실패 시나리오를 최소 3개 적고, 각 시나리오에 대해 예방/완화/복구 방법을 적는다.

### 10) 검증 절차

수용 기준을 실제로 확인하는 절차를 "명령 + 기대 결과"로 적는다. 단순 컴파일 성공을 넘어서, 변경이 실제로 효과가 있음을 입증하는 절차를 포함한다.

### 11) 놀라움과 발견

구현 중 발견한 예상 밖의 동작, 버그, 성능 상충 관계, 통찰을 기록한다. 가능하면 짧은 증거(로그/출력)를 함께 남긴다.

### 12) 의사결정 기록

작업 중 내린 결정을 아래 형식으로 기록한다.

- 결정:
  근거:
  날짜/작성자:

### 13) 결과 및 회고

완료 시점에 결과, 남은 간극, 배운 점을 요약한다. 처음의 목적과 비교해 설명한다.

### 14) 산출물과 메모

성공을 입증하는 데 가장 중요한 로그/출력/짧은 diff를 남긴다. 길게 붙여 넣기보다 핵심만 선명하게 남긴다.
