# 02. Release Checklist

## Purpose

이 문서는 `coin-zzickmock`에서 실제 릴리즈를 실행할 때 따라야 하는 체크리스트를 정의한다.
핵심 목표는 수동 배포든 자동 배포든, "무엇을 확인하고 어떤 시점에 멈춰야 하는지"를 일관되게 만드는 것이다.

## Before Release

릴리즈를 시작하기 전에 아래를 모두 확인한다.

- 대상 commit SHA가 고정되었다.
- 배포 범위가 닫혀 있다.
- 관련 PR, 계획 문서, 릴리즈 메모 초안이 연결되어 있다.
- 현재 CI가 모두 성공했다.
- 프론트 변경이 있으면 `npm run build` 결과를 확인했다.
- 프론트 Vercel 배포가 포함되면 [05-frontend-vercel-operations.md](05-frontend-vercel-operations.md)의 environment variable 계약과 Preview URL을 확인했다.
- 백엔드 변경이 있으면 `./gradlew check` 결과를 확인했다.
- 구조 변경이 있으면 `./gradlew architectureLint` 결과를 확인했다.
- 환경 변수 또는 비밀값 변경 여부를 확인했다.
- DB 스키마나 API 계약 변화가 있는지 확인했다.
- 롤백할 기준점과 담당자를 정했다.

## Stop Conditions Before Release

아래 중 하나라도 해당하면 릴리즈를 시작하지 않는다.

- 대상 commit이 바뀌고 있는데 범위가 아직 닫히지 않았다.
- CI가 실패 중이다.
- 필요한 환경 변수 주입 여부가 불명확하다.
- 스키마 변경이 있는데 호환성 계획이 없다.
- 릴리즈 노트와 롤백 기준점이 없다.

## During Release

릴리즈를 실행할 때는 아래 순서를 따른다.

1. 대상 환경과 대상 commit SHA를 다시 확인한다.
2. 필요한 경우 이전 릴리즈 기준점 또는 복구 방법을 열어 둔다.
3. 변경 범위에 맞는 순서대로 배포한다.
4. 배포 완료 직후 핵심 스모크 테스트를 수행한다.
5. 이상이 있으면 즉시 확산 여부를 판단한다.

## Smoke Checks

환경에 관계없이 최소 아래 관점은 확인한다.

- 애플리케이션이 정상 기동한다.
- 로그인 또는 핵심 진입 흐름이 깨지지 않았다.
- 프론트 주요 페이지 진입이 가능하다.
- 백엔드 핵심 API 또는 상태 점검 경로가 정상이다.
- 치명적 콘솔 에러나 서버 예외가 즉시 발생하지 않는다.
- Vercel 배포가 포함되면 deployment status가 `READY`이고, `FUTURES_API_BASE_URL`이 의도한 backend URL을 가리킨다.

변경 범위가 클수록 아래도 추가한다.

- 주문, 포트폴리오, 관심 심볼, 상점 등 주요 도메인 흐름
- 외부 연동, 알림, 캐시, 배치
- 보안 관련 경계

## After Release

릴리즈 직후 아래를 수행한다.

- 릴리즈 기록에 실제 배포 시각과 실행자를 남긴다.
- 스모크 테스트 결과를 기록한다.
- 관측해야 할 리스크를 적는다.
- 이상이 없으면 릴리즈를 완료 상태로 표시한다.
- 이상이 있으면 롤백 또는 핫픽스 여부를 결정하고 기록한다.

## Suggested Evidence

릴리즈 완료의 증거로 아래를 남기는 것을 권장한다.

- 성공한 CI 링크 또는 로그
- 배포 환경 URL 또는 식별자
- 스모크 테스트 결과 요약
- 관련 PR과 계획 문서 링크
- 롤백 기준점

## Maintenance Rule

- 실제 운영 절차가 바뀌면 이 체크리스트도 함께 갱신한다.
- 스모크 테스트가 늘어나면 항목을 빼지 말고 명시적으로 추가한다.
